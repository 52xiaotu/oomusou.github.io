Laravel 在 5.1 之後，提供了應用程式測試 (Application Testing)，讓我們不用開啟瀏覽器，就可以直接對 route、controller 與 blade 進行驗收測試，且執行速度非常快，但也因為沒有開啟瀏覽器，所以無法對 JavaScript 進行測試，若搭配了 Selenium，配合我們熟悉的 PHPUnit，就能對驗收測試加以自動化。

Motivation

Laravel 的應用程式測試非常好用，API 語意清楚，且提供 fluent 方式串接，不過 Seleinum 提供的又是另外一套 API 方式，若能將應用程式測試與 Selenium 測試整合成相同的 API，那就太棒了。

應用程式測試環境

終端機模式

Laravel 預設已經安裝好 PHPUnit，並提供捷徑在 vendor/bin 目錄下。

在專案目錄下 vendor/bin/phpunit，執行 Laravel 預設的應用程式測試。

若能看到 綠燈，表示應用程式環境已經安裝成功。1 1通常只要 PHP CLI 路徑有設定好，Laravel 正確安裝，應用程式測試就可以正常執行。

PhpStorm

PhpStorm 內建支援 PHPUnit，我們可以直接在 PhpStorm 內直接執行 PHPUnit。

PhpStorm -> Preferences -> Language & Frameworks -> PHP

  • PHP language level : 7 (return types, scalar type hints, etc.)

將 language level 調整為 7，讓 PhpStorm Code Inspection 自動針對 PHP 7 語法加以檢查。1 1關於 PhpStorm 的 Code Inspection，詳細請參考如何在 PhpStorm 使用 Code Inspection?

按 ... 設定 PHP CLI 路徑。

PhpStorm 允許我們設定多組 PHP CLI 版本供切換。

  • PHP executable : /Applications/MAMP/bin/php/php7.0.8/bin/php
  • Debugger extension : /Applications/MAMP/bin/php/php7.0.8/lib/php/extensions/no-debug-non-zts-20151012/xdebug.so

從 PhpStorm 2016.2 開始，提供了 Xdebug On Demand 功能，不必事先在 php.ini 啟動 Xdebug 影響 Composer 速度，只要事先設定好 xdebug.so 路徑，PhpStorm 會在設定中斷點除錯時，自動啟動 Xdebug。1 1關於 PhpStorm 的 Xdebug on Demand，詳細請參考如何在 PhpStorm 使用 Xdebug on Demand?

  • Interpreter : PHP7(7.0.8)

將 interpreter 設定為剛剛設定的 PHP CLI。

PhpStorm -> Preferences -> Language & Frameworks -> PHP -> PHPUnit

選擇 Use Composer autoloader :

  • Path to script : /Users/oomusou/Code/PHPConf/Laravel53SeleniumPHPUnit_demo/vendor/autoload.php

  • Default configuration file : /Users/oomusou/Code/PHPConf/Laravel53SeleniumPHPUnit_demo/phpunit.xml

Path to script 指定到專案目錄下的 vendor/autoload.php，告訴 PhpStorm 在執行 PHPUnit 時，先執行此路徑的 autoload。

Default configuration file 指定到專案根目錄下的 phpunit.xml，此為 PHPUnit 的設定檔，Laravel 已經在此檔設定好很多預設值。

選擇在專案總管的 tests/ExampleTest.php，這是 Laravel 預設所提供對 Laravel 首頁的應用程式測試範例，按滑鼠右鍵選擇 Run ExampleTest 或按熱鍵 ⌃ + ⇧ + R 執行測試。

測試結果會顯示在下方，若看到 綠燈，則表示 PhpStorm 的 PHPUnit 執行環境已經設定完成。

Selenium測試環境

phpunit-selenium

phpunit/phpunit-selenium 是由 PHPUnit 官方所提供，讓我們可以在 PHP 與 Selenium Server 溝通，直接對瀏覽器做控制，並使用 PHPUnit 做 assertion，目前僅支援 Selenium Server 2.x。1 1GitHub Commit : 安裝 phpunit-selenium

在專案目錄下由 Composer 安裝 phpunit/phpunit-selenium，由於這是開發用的套件，請加上 --dev 參數。

Selenium Server

我們必須透過 Selenium 幫我們對瀏覽器下指令，因此必須先安裝 Selenium Server。

目前 3.0 尚在 beta，且 phpunit/phpunit-selenium 目前僅支援 Selenium 2.x，所以選擇 previous release 安裝以前的版本。

選擇 2.53，這是 Selenium 2.x 的最後版本。

下載 selenium-server-standalone-2.53.1.jar，這是 2.53 的最後一個版本。

將 selenium-server-standalone-2.53.1.jar 複製到 Laravel 專案的根目錄下。

Selenium 由 Java 所開發，必須由 Java 啟動。

啟動 Selenium Server，若看到 Selenium Server is up and running，則表示 Selenium 已經正常啟動。

ChromeDriver

Selenium 預設只支援 Firefox，若要使用 Chrome，需另外安裝 ChromeDriver。

選擇目前最新版本下載。

根據平台下載不同的 ChromeDriver 版本。

將 zip 解開後，將 chromedriver 複製到 Laravel 專案的根目錄下。1 1GitHub Commit : 安裝 ChromeDriver

環境測試

當 phpunit-selenium、Selenium Server 與 ChromeDriver 都裝好後，我們就可以來對 Laravel 的預設首頁做 Selenium 測試，確認環境是否安裝成功。

啟動 PHP 內建的 web server。

會將 Laravel 預設首頁開在 http://localhost:8000/。

由 Laravel artisan 建立 test。

ExampleSeleniumTest.php1 1GitHub Commit : Selenium 環境測試

這是 PHPUnit Chapter 17.PHPUnit and Selenium 所提供的範例程式，用來對網頁做最簡單的 Selenium 測試。

第 1 行

並不是繼承 Laravel 所提供的 TestCase，而是改繼承 phpunit/phpunit-selenium 所提供的 PHPUnit_Extensions_Selenium2TestCase。

第 3 行

每個測試案例都會執行的 setUp()，可將每個測試案例都要執行的設定寫在這裡。

  • 使用 setBrowser() 設定要啟動的瀏覽器，
  • 使用 setBrowserUrl() 設定要測試網站首頁 url。

第 9 行

Laravel 預設首頁的 title 為 Laravel，我們想測試 title 是否正確。

  • 使用 url() 設定要測試的 url。
  • 使用 PHPUnit 的 assertEquals() 測試網頁的 title 是否為 Laravel。

在 PhpStorm 內執行 ExampleSeleniumTest.php，會發現 Chrome 開啟後立即結束，若能看到 綠燈，表示 phpunit-selenium、Selenium Server 與 ChromeDriver 都已經安裝成功。

第 3 行

無論是要寫單元測試、整合測試或應用程式測試，直接繼承 TestCase 即可，Laravel 已經幫我們整合了 PHPUnit、Mockery 與應用程式測試所需要的物件。

第 5 行

撰寫測試時，只要牽涉到資料庫讀寫，就必須遵造童子軍法則 : 測試結束時，要負責將假資料刪除，如此才能確保下一個測試執行時，是一個完全乾淨的資料庫，測試時才沒有任何 side effect。

每個測試案例執行完時，都會執行 tearDown()，所以可以將 truncate 動作寫在 tearDown() 內。

13 行

使用後端技術做驗收測試，最大的優點就是可以根據測試案例的需求，直接對資料庫做假資料。

這裡直接使用 Laravel 的 Model Factory。1 1關於 Model Factory，詳細請參考 Laravel 官網 Model Factories

18 行

  • visit() : 以 GET 方式對指定 URL 加以測試。
  • see() : 測試網頁是否包含指定字串，相當於 PHPUnit 的 assertContains()。
  • seeInDatabase() : 測試資料是否存在於資料庫，第 1 個參數為 table 名稱，第 2 個參數為欲驗收資料的陣列，key 為欄位名稱，value 為資料。

應用程式測試讓我們不僅可以測試網頁資料是否正確，還可以直接測試資料庫資料，也由於其 fluent API 特性，讓我們可以使用口語化的方式輕鬆閱讀測試程式碼，這也是應用程式測試最大的魅力所在。

在 一啟動顯示3筆task 的 method 內，按滑鼠右鍵選擇 Run TodoAppTest,一啟動顯示3筆task，或按熱鍵 ⌃ + ⇧ + R，只執行此測試案例。

測試會通過並顯示 綠燈，注意此時並沒有啟動瀏覽器，就如往常執行單元測試一樣，這也是應用程式測試速度超快的原因。

第 8 行

前一個測試我們使用了 Model Factory 對資料庫新增假資料，特別在測試一開始，先確認資料已經被刪除。

  • dontSee() : 測試網頁是否不包含指定字串，相當於 PHPUnit 的 assertNotContains()。
  • dontSeeInDatabase() : 測試資料是否不存在於資料庫。

14 行

  • type() : 針對 <input type=text> 做輸入，第 1 個參數為欲輸入字串，第 2 個參數為 HTML 的 name。
  • press() : 按下 <input type=submit>，第 1 個參數為 button 顯示的字串。

使用 type() 與 press() 就能模擬使用者打字輸入的動作，最後再配合 see() 與 seeInDatabase() 測試網頁與資料庫的資料。

測試會通過並顯示 綠燈，儘管這次有輸入的動作，依然沒有啟動瀏覽器。

建立 AJAX 版 Todo 專案

應用程式測試有很多優點，如執行速度快、API 語意清楚、fluent 方式串接，且還可同時測試資料庫，但由於測試時沒有啟動瀏覽器，因此無法測試 JavaScript 與 AJAX 部分，只能測試 <form> 與 submit，但實務上 JavaScript 與 AJAX 需求越來越多，顯然只有應用程式測試是不夠的。

剛剛的 Todo 專案，Add Task 有一個 <form>，而每個 Delete 都有自己的 <form>，也就是整個網頁基本上都是透過 <form> 與 submit 機制達成，因此可以順利使用應用程式測試，現在我們要將 Add Task 與所有的 Delete 都改用 JavaScript 與 AJAX 方式。

使用應用程式測試跑 AJAX 版的 Todo 就悲劇了，只有第 1 個測試案例能通過，因為後 Add Task 與 Delete 都改用了 JavaScript 與 AJAX，這就必須動用 Selenium 測試了。

Conclusion

  • 應用程式測試適合用在單純使用 <form> 來 submit 的需求，優點是速度非常快。
  • Selenium 測試適合用在使用 JavaScript 與 AJAX 的需求，由於需要開啟瀏覽器，速度稍慢。
  • 透過繼承 TestCaseSelenium，Selenium 測試也能使用與應用程式測試相同的 API，除了測試程式碼可無痛升級外，也能享受 API 語意清楚，fluent 方式串接的優點，還可以同時測試資料庫。
